# 前端现状与开发计划

面向当前前端仓库状态维护。
**最后更新**: 2026-03-21

## 1. 当前现状

### 1.1 已有基础能力

- 路由、布局、认证状态持久化、权限守卫已具备基础实现
- 配置中心页面已覆盖基础配置、组合配置、关系配置、系统配置与权限配置
- 提单页已具备复杂交互基础，包括画布、抽屉、草稿、编辑态恢复、文件校验、参数导入
- 订单列表与结果页已有基础页面和查询逻辑

### 1.2 当前核心问题

- 登录链路存在普通登录、SSO 登录、登录模式探测三套逻辑并行，需要统一入口
- 提单页承担了过多状态、取数、派生逻辑，后续维护成本高
- 提单页真实运行态已经是树状结构，但部分文档、部分接口和部分历史理解仍偏平铺
- 配置中心、提单页、结果页之间的字段语义还不够稳定
- 仍有部分历史实现与当前代码风格不一致，例如局部 API 取数方式、部分页面组件职责过重

## 2. 当前最重要的前端目标

当前前端不宜继续泛化扩展功能，优先目标应收敛为：

1. 统一登录与 SSO 入口
2. 打通配置驱动的提单主链路
3. 在不牺牲交互效率的前提下降低提单页维护复杂度
4. 保证订单详情与结果页能正确消费提单快照

## 3. 配置驱动渲染的落点

### 3.1 适合保留的技术

- React 19
- TypeScript
- TanStack Query
- Zustand
- React Hook Form + Zod
- Tailwind CSS
- React Router 7

这些技术与当前页面结构高度耦合，继续保留的成本最低。

### 3.2 建议的前端结构演进

建议围绕提单主链路继续下拆：

```text
pages/submission
  -> feature hooks
  -> schema adapters
  -> config-driven renderer
  -> snapshot serializer
```

重点是把“页面状态容器”和“配置解释逻辑”分开：

- 页面只负责用户交互编辑
- 配置解释器负责把后端配置转成可渲染字段模型
- 提交序列化器负责把页面值转成 `inputJson`、`optParam`、`submitCheck`

## 4. 登录与 SSO

当前登录链路处于收口阶段：

- 普通登录已经是业务主链路的一部分
- SSO 相关前后端代码已经存在，但还没有完全统一到同一套认证状态管理上
- 登录页已经具备“登录模式获取”和“SSO 登录交换”能力

需要明确：

- 登录 / SSO 不是提单配置化主链路的一部分
- 但它是上线 P0 并行依赖，不收口就无法稳定上线

建议：

- 前端只保留一个登录入口页
- 登录页根据后端返回的登录模式决定展示普通登录或进入 SSO
- `authStore` 不再区分“普通登录用户”和“SSO 用户”两套前端状态形态

## 5. 后续开发计划

### P0

- 统一登录模式获取、普通登录、SSO 登录的状态模型
- 收口提单页依赖的初始化配置字段和树状结构语义
- 提炼配置解释层，减少页面内硬编码判断
- 固化订单快照回填逻辑，确保编辑态、详情态一致

### P1

- 将提单页按“基础信息 / 配置解释 / 画布交互 / 提交序列化”拆分
- 增加提单主链路集成测试
- 清理页面中的历史实现残留

### P2

- 结果页复杂筛选、导出与对比
- 权限点细化到操作级
- 提单页性能优化与虚拟化细化

## 6. 评审建议

如果目标是最快上线，前端应优先接受以下原则：

- 不新增大而全能力
- 不迁移核心框架
- 先收口提单和登录
- 先统一协议，再优化界面细节
